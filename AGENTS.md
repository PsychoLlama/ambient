## Structure

- `crates/*`: Language implementation across several crates. Special cases noted below.
  - `ambient-cli/`: CLI for managing ambient codebases.
  - `ambient-platform/`: Bindings for performing effects on the host system.
  - `ambient-analysis/`: Shared parse/check/diagnose pipeline behind both `ambient check` and the LSP.
  - `ambient-lsp/`: Language server.
- `ambient.nvim/`: Neovim plugin providing basic language support.
- `tree-sitter-ambient/`: Syntax highlighting.

## Invariants

- The LSP is a renderer: anything that decides _what is an error_ or resolves a name belongs in `ambient-analysis` or the engine (`ModuleRegistry`), never in `ambient-lsp`. Do not add LSP-private indexes or resolvers.
- `crates/ambient-lsp/tests/parity.rs` pins LSP diagnostics == `ambient check` diagnostics. If it fails, fix the shared layer ā€” don't fork behavior in a frontend.
- IDE paths parse with `ambient_parser::parse_recovering` (partial AST + all errors); only the compiler uses fail-fast `parse`.
- Every item has one fully-qualified identity ā€” a real `Fqn` struct (`crates/ambient-engine/src/fqn.rs`): a `Scope` (`Builtin` = `core`, or `Workspace(pkg)`) + module path + ident path, rendered `core::primitives::number::sqrt` / `workspace::pkg::math::gcd` (`::` is the sole path separator, `.` is value/field access). The engine's resolve pass (`crates/ambient-engine/src/resolve.rs`) canonicalizes every resolved reference ā€” _same-module included_ ā€” to its `Fqn` before checking (an enum variant is the two-segment ident `[Enum, Variant]`). The checker env, native-binding tables, ability resolver, and linker key off the `Fqn` struct (via `NameKey`'s `Eq`/`Hash`) ā€” never string equivalence, and never a second lookup convention or a spelling-specific code path. Only two things stay `NameKey::Bare`: true lexical **locals**, and content-addressed method-dispatch symbols (`<uuid>::method`). Content hashes never depend on the `Fqn` ā€” `finalize_module_hashes` folds a recursive function's _bare_ short name into its group object, so the `Fqn` is a compile-time lookup key and post-hash label, never an input to content addressing. `Display` is for humans and the on-disk store only. (Registry-less checks/compiles ā€” single-file, some tests ā€” never run resolve, so their own items stay bare; a `None` module id threads that through the checker/compiler.)
- A name resolves iff something in scope put it there ā€” prelude, `use`, or a local declaration. There are no name-keyed builtin fallbacks in the checker, resolve pass, or compiler: even the containers (`List`/`Map`/`Set`) and the `Exception` ability are ordinary declarations in `core`, re-exported by the prelude (`crates/ambient-engine/src/core_lib/prelude.ab`). Type _syntax_ ā€” function arrows, tuples, `Handler<A, R>` ā€” is not a name and is out of scope for this rule: `Handler` is parsed as its own node (`CstTypeExprKind::Handler` ā†’ `Type::HandlerAnnotation`, resolved by `resolve_holes`), never an identifier-shaped type head.
- `crates/ambient-cli/tests/module_system.rs` pins the access rule: anything reachable fully-qualified works through `use`, and vice versa.
- Cross-module compile channels (imported enums, foreign variants/unit structs/const hashes/ability identities) come only from `ModuleEnv` (`crates/ambient-engine/src/module_env.rs`). Never hand-assemble them at a call site; add new channels to `ModuleEnv` so every compile sees the full view.
- The checker's import registration and `ModuleEnv::new` share the same walk functions (`module_env::imported_enum_defs`/`imported_trait_defs`). Keep them shared ā€” a private re-implementation reintroduces checker/compiler drift.
- `extern fn` is the host boundary. Declarations own signatures/docs in `.ab` source; the host binds implementations in a `NativeRegistry` keyed by (module path, name) and supplies each function's **stable UUID** ā€” the content identity. A native object encodes exactly `(uuid, param_count)`; names never enter it, so renames re-key the binding (loud build error via `verify_native_contract`, both directions) but never move a hash. Never reuse or renumber a UUID; core's assignments (`natives/core/`, block `FFFFFFFF-FFFF-FFFF-FFFE-ā€¦`) and the platform's (block `FFFFFFFF-FFFF-FFFF-FFFC-ā€¦`) are pinned by golden tests. Natives get no VM handle, no store access, no ability channel; core's are pure value transformations. Effectful natives (`stdio_out`, `fs_read`, ...) are legal but must stay module-private to `core::system` so ability default implementations are their only callers ā€” effect tracking lives at the ability layer.
- Traits are nominal too: `unique(<uuid>) trait` is mandatory, and coherence, dispatch symbols (`<type-uuid>::<trait-uuid>::<method>`), trait bounds, and operator desugaring all key off the trait uuid ā€” never the name. Operators anchor on the reserved trait uuids (`TRAIT_ADD_UUID`ā€¦ in `types/traits.rs`, block `FFFFā€¦-0010`, pinned by `validate_reserved_trait`); primitives are exempt from operator-trait dispatch (their operators are the builtin opcodes; the core impls exist only as dictionary sources).
- Trait bounds compile by **dictionary passing**, never monomorphization: a bounded item takes hidden trailing dictionary parameters (tuples of `FunctionRef`s, methods in `TraitDef::dictionary_order`). `crates/ambient-engine/src/ast/items.rs::dict_params` is the single authority on dictionary order/count ā€” the checker's scheme bounds (`Scheme::bounds`), the solver (`infer/constraints.rs`), and the compiler's slot allocation all derive from it; never compute the list a second way. Call-site dictionaries are `DictSource::Impl` (ordered impl-method symbols, hash-linked like direct calls) or `DictSource::Param` (forward the enclosing item's dictionary); every body-check site must call `finish_body_constraints` so pending constraints never leak across bodies. Content hashes gain no new channels from bounds ā€” dictionaries are ordinary bytecode. Ability-method bounds enter the canonical signature as `bound:<idx>:<Name>` pseudo-params (spelled name, by design ā€” like `named:Duration`).
- Abilities are nominal: the mandatory `unique(<uuid>)` prefix is the identity (`AbilityId::from_uuid`), so renames/moves never change it and same-shaped declarations never unify. A method's identity is its `MethodKey` (`ambient-core/src/identity.rs`): blake3 of ability uuid + canonical signature + default-implementation hash ā€” the method _name_ is excluded (rename-stable), the body is included (same-signature methods differ by behavior; edits re-key loudly). Every method carries a default implementation (the body an unhandled perform calls as a plain function); the abstract carve-out is never-returning methods (`: !`), which may omit the body ā€” an unhandled abstract perform is a runtime fault. Never methods are catch-only: the perform unwinds (the VM discards the delimited continuation ā€” the `never` flag on `AbilityMethodRef`/`SuspendedAbility` carries this to runtime), handler arms cannot `resume`, and a `!`-typed expression adopts any context type in the checker (bottom elimination via `adopt_never`; introduction stays strict). The prelude's `Exception::throw` is the canonical never method, anchored by `ambient_core::exception::EXCEPTION_UUID` and reproduced by `core_lib/exception.ab`. A body's allowed effects are the ability's declared `with` row, which is also what keeps a method's implementation hash from depending on itself. Default implementations compile under `<ability-uuid>::<method>` dispatch symbols and flow through the ordinary linking table ā€” never add a parallel channel. Canonical signature rendering still goes through `ability_id_infer`'s seeded context (`infer/check/abilities.rs`); a fresh `Infer::new()` would corrupt method keys. The golden test pins the platform ability ids (reserved block `FFFFFFFF-FFFF-FFFF-FFFD-ā€¦`).

## Committing

- Run `just check` and keep it passing.
- `just check` enforces per-file line budgets (`scripts/file-size-budgets.txt`): new `.rs` files cap at 1000 lines; grandfathered files may only shrink. Split a file instead of growing it ā€” never add a budget entry for new growth.
